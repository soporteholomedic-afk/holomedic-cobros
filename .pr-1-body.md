## Body

## Summary

Exposes the medical-fitness verdict (`Condic`) returned by `SP_RPT_MATRIZICCGSA` as a new `Aptitud` column on `WorkerDetailTable`. The underlying field already travels on the wire (it's part of the SP resultset), but the client-side merge in `useUnifiedResults` was discarding it. This PR adds the type, the normalization helper, the hook capture, and the 9th column rendering — pure additive change, no new dependencies, no infra.

## Spec Acceptance Criteria (REQ-WT-1, REQ-WT-3, REQ-WT-4)

- [x] **AC-1**: A primary row whose `Condic` is `'APTO'` shows `APTO` in the `Aptitud` cell. (Test: `useUnifiedResults` scenario 12; `WorkerDetailTable` `should render condic 'APTO' verbatim`.)
- [x] **AC-2**: A primary row whose `Condic` is `'NULL'` shows `—` (`\u2014`) in the `Aptitud` cell. (Test: `useUnifiedResults` scenario 13 — `'NULL' | 'null' | 'Null' | '   '` → `''`, then `WorkerDetailTable` `should render em-dash in the Aptitud cell when condic is empty`.)
- [x] **AC-3**: A primary row whose `Condic` is `'Null'` (mixed case) shows `—` in the `Aptitud` cell. (Test: `useUnifiedResults` scenario 13.)
- [x] **AC-4**: A sub-row whose `Condic` is `'NO APTO'` shows `'NO APTO'` even if the primary row's `Condic` is empty. (Test: `useUnifiedResults` `should give each sub-ficha its own Condic from the corresponding worker row`; `WorkerDetailTable` `should show each sub-ficha its own condic when expanded`.)
